

**PROJECT NAME**
Multi-Tenant AI Form Builder & Analytics Platform


**COURSE:** BLM 4062 Research Project II — Ankara University

---

**WHAT IS THIS PROJECT**

A SaaS (Software as a Service) web platform where organizations (called "workspaces") can create forms, collect responses, and analyze them using AI. The key difference from tools like Google Forms is that this platform uses AI to generate questions automatically, and uses vector embeddings to semantically analyze and cluster free-text responses — meaning it understands meaning, not just keywords.

---

**HOW THE PLATFORM WORKS (user flow)**

1. A user registers and creates a workspace
2. Inside the workspace they create a form — either manually or by describing the purpose and letting AI generate the questions
3. The form is published and shared via a public link or embed code
4. Respondents fill out the form (no login required)
5. Each free-text response is automatically embedded into a 1536-dimensional vector and stored
6. The workspace owner opens the analytics dashboard and sees: response count, completion rate, average duration, sentiment distribution (positive/negative/neutral), and semantic clusters of similar responses
7. Optionally, a webhook fires an HTTP POST to an external URL every time a new response arrives

---

**TECH STACK**

- Frontend: Next.js 16 with App Router, TypeScript, TailwindCSS
- Database: Supabase (PostgreSQL 15) with pgvector extension
- Auth: Supabase Auth with JWT and cookie-based sessions
- LLM: Groq API — model llama-3.3-70b-versatile — for question generation and sentiment scoring
- Embeddings: OpenAI API — model text-embedding-3-small — produces 1536-dimensional vectors
- Streaming: SSE (Server-Sent Events) for real-time AI question streaming into the editor
- Deployment: Vercel (frontend + API routes)
- Security: Row-Level Security (RLS) on every database table, HMAC-SHA256 signed webhooks

---

**DATABASE TABLES (all 6)**

1. workspaces — id, name, owner_id, plan_tier, created_at
2. workspace_members — workspace_id, user_id, role (owner/editor/viewer), joined_at
3. forms — id, workspace_id, title, description, schema (JSONB), logic_graph (JSONB), is_published, created_at, updated_at
4. responses — id, form_id, answers (JSONB), respondent_meta (JSONB), submitted_at
5. embeddings — id, response_id, form_id, question_key, embedding (vector 1536), sentiment_score, sentiment_label, created_at
6. webhooks — id, workspace_id, target_url, event_type, secret_key, is_active, created_at

Every table has Row-Level Security enabled. Users can only access data that belongs to their workspace. This is enforced at the database level, not just in application code.

---

**KEY TECHNICAL CONCEPTS**

RAG (Retrieval-Augmented Generation): When generating form questions, the AI is not just given a blank prompt. It receives the form purpose, target audience, desired question count, and supported question types as context. This produces domain-specific, relevant questions rather than generic ones.

pgvector and semantic clustering: Free-text answers are converted to vectors using OpenAI embeddings. These vectors are stored in PostgreSQL via the pgvector extension. The analytics dashboard uses cosine similarity (the <=> operator) and k-means clustering with k = square root of total responses to group semantically similar answers together — even if they use different words.

DAG conditional logic: Forms support branching — "if the user answers X, show question Y, otherwise show question Z." This is modeled as a Directed Acyclic Graph where each question is a node and each conditional rule is an edge. The client-side traversal uses Depth-First Search to decide which question to show next.

JSONB schema: Form questions are not stored as fixed columns. They are stored as a flexible JSON array in the schema column. This means new question types can be added without changing the database structure.

Multi-tenancy: Every piece of data is scoped to a workspace. RLS policies ensure that even if a bug exists in application code, a user's database queries will never return another workspace's data.

---

**EVALUATION METRICS (from Design Document)**

The project will be evaluated on 4 dimensions:

1. LLM question quality: Each AI-generated question set is scored 1-5 on scope, clarity, diversity, and audience fit. Compared against manually designed forms.

2. Embedding cluster quality: Measured by Silhouette Score. Formula: S(i) = (b(i) - a(i)) / max(a(i), b(i)). Target: average score above 0.5.

3. Performance targets:
   - Form response submit: under 500ms
   - Embedding generation (async): under 3000ms
   - Dashboard load: under 1000ms
   - SSE first token: under 800ms

4. RLS security: Cross-workspace access attempts must be rejected at the database level. SQL injection attempts must fail.

PHASE 1 — Foundation (COMPLETE)

PHASE 2 — Database Schema (COMPLETE)
- All 6 tables running in Supabase
- pgvector(1536) column on embeddings table
- IVFFLAT index on embedding column for cosine similarity queries
- Auto-trigger: workspace owner is automatically added to workspace_members on creation
- RLS policies tested and active on all tables
- updated_at trigger on forms table

PHASE 3 — Form Engine (IN PROGRESS)

Completed so far:
- New form page at /dashboard/[workspaceId]/forms/new — title input, description, create button, AI placeholder button
- Form editor page at /dashboard/[workspaceId]/forms/[formId] — top bar with back/preview/publish, form metadata display
- Manual question editor — supports short_text, long_text, multiple_choice, rating, and file placeholder question types
- Form schema auto-save to Supabase
- Publish toggle and public preview/share link flow
- Shared form engine utilities for schema normalization, DAG logic graph normalization, cycle detection, visible-question traversal, and answer filtering
- Conditional display rules in the editor — later questions can be shown only when an earlier answer equals a configured value
- Public form response page at /f/[formId] — no authentication required, renders published forms, applies conditional logic, validates visible required questions, and submits answers to the responses table
- Responses management page at /dashboard/[workspaceId]/forms/[formId]/responses — workspace members can view submitted answers in a table
- CSV export route at /dashboard/[workspaceId]/forms/[formId]/responses/export — downloads submitted answers with readable question and option labels

Still to build in Phase 3:
- Full file upload support for file questions
- XLSX export if required beyond CSV
- Embed code generation
- Focused tests for schema parsing, DAG traversal, cycle detection, and visible-answer submission

PHASE 4 — AI Module (COMPLETE)

- Groq SSE endpoint at /api/ai/generate — receives form purpose, target audience, question count, question types. Sends RAG-style prompt to Groq llama-3.3-70b-versatile. Streams questions back using Server-Sent Events. Questions appear in the form editor in real time.
- Wire up the "Generate with AI" button in the new form page to call this endpoint
- OpenAI embedding pipeline — after a response is submitted, extract free-text answers, send to OpenAI text-embedding-3-small, store the 1536-dim vector in the embeddings table. This runs asynchronously so it does not slow down form submission.
- Groq sentiment scoring — after embedding is stored, send the text to Groq and classify it as positive, negative, or neutral. Update the sentiment_label and sentiment_score fields in the embeddings table.

PHASE 5 — Analytics Dashboard (COMPLETE)

- Response metrics at /dashboard/[workspaceId]/forms/[formId]/analytics — total responses, completion rate, average time to complete, sentiment breakdown chart
- Semantic clustering — query embeddings using pgvector cosine distance operator <=>, run k-means with k = square root of response count, display clusters of similar answers grouped together
- Charts using Recharts or Chart.js — sentiment pie chart, response over time line chart, cluster visualization
- Silhouette Score calculation and display to validate cluster quality

PHASE 6 — Integrations and Testing (NOT STARTED)

Payload is signed with HMAC-SHA256 using the webhook's secret_key. Signature is sent in X-Webhook-Signature header.
- Webhook management UI — add/delete/toggle webhooks from the workspace settings page
- Unit tests per module: form engine JSONB parsing, DAG DFS traversal and loop detection, embedding vector dimension check, webhook HMAC signature verification, RLS cross-workspace rejection
- Integration test: full flow from register → workspace → LLM form generation → form response → embedding → dashboard
- Performance benchmarks: measure all 4 performance targets using browser dev tools and Supabase query analyzer
